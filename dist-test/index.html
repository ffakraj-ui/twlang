Compiler pipeline, diagnostics, aur smoke checks ek hi panel mein.

Yeh reusable block website showcase ke liye nahi, balki framework testing ke liye hai. Ismein sample input, expected compiler phases, aur error-report shape ko direct review kiya ja sakta hai.

Input fixture

Small `.tw` sample jo parser, semantic checks, aur renderer tino touch karta hai.

            
page { title "Lab" layout "main" render static }
let heading = "Compiler"
BODY {
  section {
    class "panel"
    h2 "Compiler heading"
  }
}

Expected phases

01 tokenize
02 parse
03 analyze
04 lower
05 render

Diagnostics shape

Ab structured output mein severity, code, line, column, phase, aur suggestion sab include kiya ja sakta hai.

error • TW1000

Syntax ya parser failure ko exact phase ke saath pakdo.

warning • TW2001

Missing title, undefined symbols, aur weak authoring hints ko alag rakho.

summary

CLI check output mein total counts aur phase-wise grouping inspect karo.

Goal simple hai: website polish se zyada framework observability. Agar parser, AST, IR, ya renderer fail kare, test page se immediately trace milna chahiye.

Fixture snippets

Testing ke liye small inputs aur expected behavior side by side.

Yeh snippets intentionally concise rakhe gaye hain taaki parse failure, semantic issues, aur successful render tino quickly reproduce kiye ja sakein.

Passing fixture

            
page { title "Ok" layout "main" render static }
BODY { p "ready" }

Expected: no errors, HTML output available, summary errors `0`.

Symbol warning

            
page { title "Warn" layout "main" render static }
BODY { p "missingName placeholder" }

Expected: real warning case ko reproduce karte waqt placeholder symbol manually add karo, aur diagnostic entry mein code plus source location inspect karo.

Syntax break

            
page { title "Broken" layout "main" render static }
BODY { p "oops }

Expected: tokenizer or parser phase failure with structured suggestion fields.
